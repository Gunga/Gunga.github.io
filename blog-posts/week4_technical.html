Do I need an Array or a Hash?

by Joan Torres

To think there used to be a time when we thought data was a simple thing. I'm gonna go right ahead a let you all in on a secret: it's not. At least we made sure it didn't stay simple. The way we collect and organize data has evolved to such a point that we have different data structures to deal with certain data. There are simple ones and there are advanced ones. In this entry, we'll take a look at some harmless, simple ones, the Array and the Hash.

The Array

The arrays allow us to have a collection of elements which are indexed by integers. Storing data and selecting data from an array is usually done in constant time as long as we know the index. As indexing begins with 0, if we wanted to store or read the fourth element of an array we'd use number 3 as the index. Iterating over arrays is common when we don't know an element's location. The biggest drawback arrays can present are in programming languages where only static arrays can be initialized. This means the array has a limited size to store data and cannot grow once full. Ruby's arrays seem to be dynamic arrays. Which allow us to push elements to the end of the array without worrying about the array filling up.

The Hash

The hash is similar to the array. The main purpose of the hash is also to find and store data in constant time. However, the index of the hash is not a number, it is a key. This key could be anything from a name to a unique_id. Therefore, our key is very important to finding a value. Otherwise, we'd have to iterate over the hash to find the value we're looking for which won't make it much different form other similar data structures. Ruby's hash is pretty similar to what many call an associative array. This means that our collection of elements is stored using "key & value" pairs instead of integers that serve as the index to the values.

Which should I use?

Well, when it comes down to it, their usage mostly depends on the application. When it comes to linear data, having an array could be more beneficial than having a hash if there's too little data to store, because the hash has to use a "hash function" to reach the location of our value from the key. Running the hash function may use more resources than we save from reaching our data quickly if the amount of data is too little. However, when working with table data linearly the hash is usually more intuitive to work with. Not just that, but it's easier to add a column to a table without having to change as much code as we may have to with an array.

Let's take a look at this table

name last_name gpa
Jane Smith 88
Johnny Jackson 90

If we read one row and use an array to store it we will have something like this:

row = [jane, smith, 88]

Therefore, we can access data from the array using numbers:

row[0]=jane
row[1]=smith
row[2]=88

However, if that table changes into something such as this:

student_id name last_name session_id gpa
111-06-5212 Jane Smith 010 88
111-08-1429 Johnny Jackson 046 90

Then row[1] would be where the name is stored instead of row[0]. Therefore, all of our code now needs to be updated where an indexed reference to our row array is made so that we can refer to the actual values we wanted. Whereas if we had used a hash and read the row in the table the first time we would have gotten this:

row = { "name" => "Jane", "last_name" => "Smith", "gpa" => "88"}

the second time, this:

row = { "student_id" => "111-06-5212", "name" => "Jane", "last_name" => "Smith", "session_id" => 010, "gpa" => "88"}

and on BOTH cases row["name"] would refer to "Jane". Therefore, adding those new columns to a table our application has to read wouldn't mean changing every reference to row in the code.

It is very important to understand the application we're designing in order to determine which data structure would be more suited for it. When it comes to multi-dimensional data a combination can be used: array of arrays, hash of hashes, array of hashes , hash of arrays, and so on. The important thing here understand each well enough to determine which is the better choice in the long wrong when it comes to performance and maintainability.
